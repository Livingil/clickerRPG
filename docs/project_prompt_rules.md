# Project Prompt Rules (UTF-8 / Russian Text Safety)

Используй этот блок как обязательные правила для любых правок в проекте.

## Core Rules

1. Всегда сохраняй все текстовые файлы в `UTF-8` (предпочтительно `UTF-8 without BOM`, если инструмент поддерживает).
2. Никогда не вставляй русский текст в неправильной кодировке (`Windows-1251`, `ANSI`, и т.п.).
3. Любой русский текст в `.gd`, `.tscn`, `.tres`, `.md`, `.json`, `.csv` должен отображаться читабельно, без крокозябр.
4. Если видишь битый текст вида `РџСЂРёРІРµС‚` — это ошибка кодировки. Исправляй до завершения задачи.
5. Не смешивай кодировки в одном файле. Один файл = одна кодировка (`UTF-8`).

## Editing Rules

1. Перед правкой файлов с русским текстом проверяй, что редактор/скрипт пишет в `UTF-8`.
2. При программной перезаписи файлов всегда указывай `UTF-8` явно.
3. Для массовых замен по проекту сначала делай точечную проверку 1-2 файлов на читаемость русского текста.
4. Не использовать инструменты/команды, которые могут молча перекодировать файл в локальную ANSI-кодировку.

## Verification Checklist (обязательно перед завершением)

1. Проверить измененные файлы с русским текстом визуально: нет крокозябр.
2. Проверить, что строки RU/EN в UI читаются нормально в исходниках.
3. Если менялись локализации, открыть соответствующие блоки и убедиться, что русский текст не поврежден.
4. Если есть сомнения по кодировке, перезаписать файл в `UTF-8` и повторно проверить.

## Failure Policy

1. Если обнаружен битый русский текст — задача не считается завершенной.
2. Сначала исправить кодировку/текст, только потом продолжать функциональные правки.
3. При любом конфликте приоритет: корректный русский текст и `UTF-8` выше всех остальных изменений.

## DRY Rules

1. Соблюдай `DRY` (Don’t Repeat Yourself): не дублируй одинаковую бизнес-логику в нескольких местах.
2. Если один и тот же расчет/маппинг/ветвление повторяется 2+ раза, выноси в:
   - общую функцию, или
   - константную таблицу (`Dictionary`/config), если это данные баланса.
3. Не копируй длинные `match/if`-блоки между файлами — делай единый источник правды.
4. Для балансных параметров предпочитай data-driven подход (таблицы коэффициентов), а не hardcode по месту использования.
5. DRY не должен ухудшать читаемость: избегай абстракций «ради абстракций».

## DRY Checklist (перед завершением)

1. Нет ли дублирующих формул в измененных файлах.
2. Нет ли повторяющихся списков/коэффициентов, которые уже есть в проекте.
3. Есть ли единая точка правки для новых балансных данных.

## Ready-to-Use Prompt Block

```text
Follow strict encoding rules for this repository:
- Save all text files as UTF-8 (preferably UTF-8 without BOM).
- Never introduce mojibake/corrupted Russian text.
- Any Russian strings must stay readable Cyrillic, not sequences like "РџСЂ...".
- If corrupted Russian text appears, fix it before finishing the task.
- After edits, re-check changed files containing Russian text for encoding integrity.
```
